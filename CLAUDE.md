# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

NearbyFinder is an iPhone treasure-hunt app built on Nearby Interaction (UWB): one iPhone is hidden, the other finds it using live distance/direction readings. Real functionality requires two physical UWB-capable iPhones (iPhone 11+, not SE); Xcode can also simulate NI between two booted simulators.

- Xcode project: `NearbyFinder.xcodeproj`, targets `NearbyFinder` (iPhone-only: `SUPPORTED_PLATFORMS = iphoneos iphonesimulator`, `TARGETED_DEVICE_FAMILY = 1` — deliberately restricted for release since iPad/Mac/Vision have no UWB) and `NearbyFinderWatch` (watchOS companion, embedded with `platformFilter = ios`)
- Deployment target: iOS/watchOS 26.5; bundle IDs `jp.hibiki.NearbyFinder` / `.watchkitapp`
- The `#if os(iOS)` / `#else` stubs in the managers predate the platform restriction; they are dead code on the only remaining platform but kept as guards. Note `canImport(NearbyInteraction)` is NOT a sufficient platform check — the module imports on macOS with unavailable APIs.
- No test targets yet. No Swift Package dependencies.

## Commands

Build for iOS Simulator (Debug):

```sh
xcodebuild -project NearbyFinder.xcodeproj -scheme NearbyFinder \
  -destination 'platform=iOS Simulator,name=iPhone 17' build
```

Build the watch app alone: `-scheme NearbyFinderWatch -destination 'generic/platform=watchOS Simulator'`.

Run tests (once a test target is added): replace `build` with `test`; run a single test by appending `-only-testing:<TestTarget>/<TestClass>/<testMethod>`.

There is no linter configured.

## Architecture

Layering, bottom to top:

- `MultipeerSession.swift` — thin MC wrapper that advertises and browses simultaneously with service type `nearbyfinder`. Normally only the peer with the larger `displayName` invites; after ~6 s the other side may invite as a one-way-discovery fallback. Pending invitations are never overlapped. A jittered retry loop refreshes discovery and rebuilds the transport after repeated failures. Because iOS 26 can remain in `.connecting` without delivering either terminal callback, a 15 s watchdog rebuilds the entire transport. Connected sessions exchange a filtered 1-byte ping every 5 s and rebuild after 15 s of silence; reliable-send failures also trigger recovery after three consecutive errors. Recovery must rebuild the transport and notify `onPeerDisconnected`, never rely on bare `session.disconnect()`. Every delegate callback compares the callback object's identity with the current session/browser/advertiser so queued events from a torn-down transport cannot corrupt new state. `refreshConnection()` audits local versus framework state on foreground return. Both devices must run the same build. `os.Logger` category `mc` records transitions, watchdogs, stale callbacks, and rebuilds.
- `NearbySessionManager.swift` — `ObservableObject` owning the player `NISession`; publishes `status`, distance, and direction data. Each NI session gets a UUID. `.discoveryToken(sessionID:data:)` is resent every ~4 s until the peer applies it and responds with `.discoveryTokenAck(tokenSessionID:senderSessionID:)`; `.ranging` is published only after this ACK and successful application of the peer token, preventing one-sided ready state. ACKs and tokens from old NI generations are ignored, while an ACK that arrives before its peer token is held until the matching generation arrives. The peer token is retained and reapplied when only the local NI session is recreated. All NI delegate callbacks verify that their session is still current, so delayed invalidation/removal callbacks from `stop()` cannot restart or overwrite a new session. CRITICAL: `.peerEnded` must never invalidate the healthy local session or reuse the dead peer token; clear the peer token/generation and wait for a fresh one. `refreshConnectionAfterForeground()` audits MC, reruns the live NI configuration, and re-announces the token. Permission denial must not restart in a loop. Camera assistance and the shared `ARSession` remain required for modern-iPhone direction and AR rendering. `os.Logger` category `nearby` records token generations, ACKs, config runs, and lifecycle recovery.
- `GameManager.swift` — game state machine (`lobby → hiding → hunting → finished`) plus `ProximityFeedback` (distance-driven haptic pulses synced with sonar pings). Roles sync over `.roleSelected` (a random `priority` breaks the both-picked-the-same-role tie). Either player's one tap starts BOTH devices — a deliberate no-host design (2 players in the same room negotiate verbally; don't add a host/ready-check flow). The abruptness on the passive side is instead softened by `RoleRevealView`: `isRevealingRole` covers the first ~2 s of hiding on both devices with an「あなたは○○」announcement — keep it. (A second `.chase` mode existed briefly — 逃走中, distance-based auto-capture — but was removed as unwanted; see git history if it ever comes back.) Hide/hunt durations are lobby-configurable, persisted in UserDefaults, and synced over `.settingsChanged` (last-write-wins); `.roleSelected` also carries the initiator's settings, which are authoritative for that round. Because each device persists its own settings (which can diverge across sessions/partners), the designated leader (larger MC displayName, via `nearby.onConnected` → `isDesignatedLeader`) broadcasts its settings once on connection so both lobbies show the same values. Settings are editable ONLY in the lobby — a title-screen settings sheet was added and then removed (the user doesn't want the same settings on two screens); don't reintroduce it. The hunt has a time limit (`huntDuration`, default 300 s); the *treasure* device is authoritative for timeout — its watchdog sends `.timeUp` and both finish with `outcome = .treasureWon`. Rounds form a best-of series (`seriesTarget` = first to 3): round wins are counted locally from (outcome, own role) so both devices agree without extra messages, and `.rematch` swaps roles and starts the next round (guarded by `phase == .finished`, which makes simultaneous rematch taps converge). Returning to the lobby resets the series. Found is confirmed physically, not by distance: the hunter uses the `SlideToConfirmButton` on the *treasure* device's screen — a phone-call-answer-style slider, slide the thumb to the end with no hold delay (`confirmFound()`, treasure-side only), which sends `.found`. The near-full-width directional slide is the accidental-touch protection; don't replace it with a tap or long-press button (fabric/skin contact triggers those), and don't add a hold-to-arm delay back (tried; it felt unresponsive — the user explicitly chose the pure slide). The two-stage gesture exists because the hidden phone may be pressed against fabric/skin; don't replace it with a plain tap or long-press. Distance-based auto-detection was removed deliberately — proximity fires before the phone is physically located. Win counts and best clear time (`GameStats`) persist per device in UserDefaults. During hunting it records `distanceHistory` (1 sample/sec, both devices) for the Swift Charts approach graph in `ResultView`, and relays phase/role/deadline/outcome to the paired Watch via `syncWatch()` on every transition. It re-publishes the nested `NearbySessionManager`'s `objectWillChange` — nested `ObservableObject`s don't propagate automatically, so views observing `GameManager` still update on distance changes.
- `GameAudio.swift` — sounds are synthesized at runtime with AVAudioEngine (sine-wave ping / fanfare / time-up buffers); there are no audio asset files. Uses the `.ambient` session category on purpose: respects the silent switch and mixes with other audio.
- Views: `ContentView.swift` (title screen + phase switch + `LobbyView`/`HidingView`/`TreasureWaitView`/`ResultView`) and `HuntingView.swift` (Find-My-style finding UI: black background, big arrow when NI provides `direction`, remaining-time header, green fade + pulse rings under 1 m). `TitleView` gates everything: `game.start()` (radios, permission prompts) and `isIdleTimerDisabled = true` (keeps the hidden phone ranging) only run when the user taps はじめる — don't move them back to launch. The lobby's タイトルへ button reverses this via `game.stop()` → `NearbySessionManager.stop()` → `MultipeerSession.stop()`; the MC stop must rebuild a fresh transport (torn-down advertiser/browser have their delegates detached and can't be restarted), which is why `stop()` shares `resetTransport()` with the failure-recovery `rebuildTransport()`. It shows persisted `GameStats`, a how-to-play sheet, and disables start via `nearby.isDeviceSupported` (checkable before `start()`, false in the stub).
- Apple Watch support (`NearbyFinderWatch/`, hunter-side accessory display): watchOS has no MultipeerConnectivity, so tokens relay Watch → WatchConnectivity (`PhoneWatchRelay` on iOS, applicationContext keys `watchToken`/`peerToken`) → own iPhone → MC (`.watchToken`/`.watchPeerToken` messages) → other iPhone, which opens a SECOND `NISession` (`watchPeerSession`, minimal `WatchPeerSessionDelegate`) against the watch and replies with its token. UWB ranging then runs directly Watch ↔ other iPhone; `WatchRanger` publishes distance (watchOS NI provides distance only, never direction) and `WatchHuntView` renders it with wrist haptics. Game state (phase/role/deadline/outcome) is relayed in the same applicationContext — note `updateApplicationContext` REPLACES the whole dictionary, so `PhoneWatchRelay` keeps one merged `outgoingContext` (token + state) and always sends it whole. The watch hides distance outside the hunting phase and shows the outcome with a haptic when the game ends. The watch app suspends on wrist-down — ranging resumes on wrist raise.

## Constraints and conventions

- The build uses `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` (Xcode 26 template). Framework delegate callbacks (MCSessionDelegate, NISessionDelegate) arrive on non-main queues, so delegate conformances are declared `nonisolated` and hop to the main actor via `Task { @MainActor in ... }`. Follow this pattern for any new delegate-based code.
- Info.plist: `GENERATE_INFOPLIST_FILE = YES` is combined with a checked-in `Info.plist` at the repo root (`INFOPLIST_FILE = Info.plist`), which holds the keys that can't be expressed as `INFOPLIST_KEY_*` build settings — notably the `NSBonjourServices` array (`_nearbyfinder._tcp/_udp`, required for MC browsing) plus the NI/local-network usage descriptions. It lives *outside* the `NearbyFinder/` folder on purpose: that folder is a `PBXFileSystemSynchronizedRootGroup`, and a plist inside it would be bundled as a resource and conflict.
- NI peer sessions suspend when either app leaves the foreground; both phones must keep the app frontmost.
