## Project Overview

NearbyFinder is an iPhone treasure-hunt app built on Nearby Interaction (UWB): one iPhone is hidden, the other finds it using live distance/direction readings. Real functionality requires two physical UWB-capable iPhones (iPhone 11+, not SE); Xcode can also simulate NI between two booted simulators.

- Xcode project: `NearbyFinder.xcodeproj`, targets `NearbyFinder` (iPhone-only: `SUPPORTED_PLATFORMS = iphoneos iphonesimulator`, `TARGETED_DEVICE_FAMILY = 1` ŌĆö deliberately restricted for release since iPad/Mac/Vision have no UWB) and `NearbyFinderWatch` (watchOS companion, embedded with `platformFilter = ios`)
- Deployment target: iOS/watchOS 26.5; bundle IDs `jp.hibiki.NearbyFinder` / `.watchkitapp`
- The `#if os(iOS)` / `#else` stubs in the managers predate the platform restriction; they are dead code on the only remaining platform but kept as guards. Note `canImport(NearbyInteraction)` is NOT a sufficient platform check ŌĆö the module imports on macOS with unavailable APIs.
- No test targets yet. No Swift Package dependencies.

## Commands

Build for iOS Simulator (Debug):

```sh
xcodebuild -project NearbyFinder.xcodeproj -scheme NearbyFinder \
  -destination 'platform=iOS Simulator,name=iPhone 17' build
```

Build the watch app alone: `-scheme NearbyFinderWatch -destination 'generic/platform=watchOS Simulator'`.

Run tests (once a test target is added): replace `build` with `test`; run a single test by appending `-only-testing:<TestTarget>/<TestClass>/<testMethod>`.

There is no linter configured.

## Architecture

Layering, bottom to top:

- `MultipeerSession.swift` ŌĆö thin MC wrapper using service type `nearbyfinder`. Bonjour `discoveryInfo` and invitation context carry protocol version 3 plus a per-launch instance UUID; incompatible peers are rejected before MC connects. All compatible discoveries are recorded, but only one connection attempt runs at a time: the candidate is locked when an invite is sent or accepted and unlocked when the attempt ends. CRITICAL (real-device only): AWDL caches Bonjour ads of terminated launches for minutes, and those ghosts are often reported before the live peer ŌĆö locking the first-found peer permanently and rejecting everyone else was a livelock where devices never connected while simulators (no stale cache) worked. So: an invite that ends without ever reaching `.connecting` after Ōēź6 s (or whose state callback never arrives) blocklists that instanceID for ~25 s and the next candidate is tried; an incoming invitation is proof of life (clears the blocklist entry), and if the current attempt is stalled (no `.connecting`, or `.connecting` older than 4 s) it is abandoned via session replacement to accept the live invitation. The `MCPeerID` stays stable across transport rebuilds within the process so stale advertisements do not look like extra devices. Normally only the peer with the larger `displayName` invites; after ~6 s the other side may invite as a one-way-discovery fallback. If both invite simultaneously, the leader's send wins only while fresh (<4 s; a stale attempt no longer exists on the peer's side), and the non-leader abandons its outgoing `MCSession` and accepts on a fresh session so a delayed cancellation callback cannot tear down the established connection. A jittered retry loop drives re-invites/candidate rotation, refreshes discovery only when no eligible candidate exists (AWDL throttles frequent restarts), and rebuilds the transport after repeated handshake failures. Because iOS 26 can remain in `.connecting` without delivering either terminal callback, a 15 s watchdog rebuilds the entire transport. Connected sessions exchange a filtered 1-byte ping every 5 s and rebuild after 20 s of silence; inactive/background time and delayed task scheduling receive a 20 s foreground grace period. Reliable-send failures also trigger recovery after three consecutive errors. Recovery must rebuild the transport and notify `onPeerDisconnected`, never rely on bare `session.disconnect()`. Every delegate callback compares the callback object's identity with the current session/browser/advertiser so queued events from a torn-down transport cannot corrupt new state. `refreshConnection()` audits local versus framework state on foreground return. Both devices must use protocol-compatible builds. `os.Logger` category `mc` records transport generation, attempt ID, peer selection, framework error domain/code, lifecycle transitions, watchdogs, stale callbacks, and rebuilds.
- `NearbySessionManager.swift` ŌĆö `ObservableObject` owning the player `NISession`; publishes `status`, distance, and direction data. Each NI session gets a UUID. `.discoveryToken(sessionID:data:)` is resent every ~4 s until the peer applies it and responds with `.discoveryTokenAck(tokenSessionID:senderSessionID:)`; `.ranging` is published only after this ACK and successful application of the peer token, preventing one-sided ready state. ACKs and tokens from old NI generations are ignored, while an ACK that arrives before its peer token is held until the matching generation arrives. The peer token is retained and reapplied when only the local NI session is recreated. All NI delegate callbacks verify that their session is still current, so delayed invalidation/removal callbacks from `stop()` cannot restart or overwrite a new session. CRITICAL: `.peerEnded` must never invalidate the healthy local session or reuse the dead peer token; clear the peer token/generation and wait for a fresh one. `refreshConnectionAfterForeground()` audits MC, reruns the live NI configuration, and re-announces the token. Permission denial must not restart in a loop. Camera assistance and the shared `ARSession` remain required for modern-iPhone direction and AR rendering. `os.Logger` category `nearby` records token generations, ACKs, config runs, and lifecycle recovery.
- `GameManager.swift` ŌĆö game state machine (`lobby ŌåÆ hiding ŌåÆ hunting ŌåÆ finished`) plus `ProximityFeedback` (distance-driven haptic pulses synced with sonar pings). Roles sync over `.roleSelected` (a random `priority` breaks the both-picked-the-same-role tie). ## Constraints and conventions

- The build uses `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` (Xcode 26 template). Framework delegate callbacks (MCSessionDelegate, NISessionDelegate) arrive on non-main queues, so delegate conformances are declared `nonisolated` and hop to the main actor via `Task { @MainActor in ... }`. Follow this pattern for any new delegate-based code.
- Info.plist: `GENERATE_INFOPLIST_FILE = YES` is combined with a checked-in `Info.plist` at the repo root (`INFOPLIST_FILE = Info.plist`), which holds the keys that can't be expressed as `INFOPLIST_KEY_*` build settings ŌĆö notably the `NSBonjourServices` array (`_nearbyfinder._tcp/_udp`, required for MC browsing) plus the NI/local-network usage descriptions. It lives *outside* the `NearbyFinder/` folder on purpose: that folder is a `PBXFileSystemSynchronizedRootGroup`, and a plist inside it would be bundled as a resource and conflict.
- NI peer sessions suspend when either app leaves the foreground; both phones must keep the app frontmost.
