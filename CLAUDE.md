## Project Overview

NearbyFinder (product/display name: **Treasure Radar** — set via `INFOPLIST_KEY_CFBundleDisplayName` on both targets and used in user-facing strings; the project and targets keep the NearbyFinder name) is an iPhone treasure-hunt app built on Nearby Interaction (UWB): one iPhone is hidden, the other finds it using live distance/direction readings. Real functionality requires two physical UWB-capable iPhones (iPhone 11+, not SE); Xcode can also simulate NI between two booted simulators — but simulator NI does NOT exercise the real UWB/AWDL/ARKit paths (see the rebuild note below).

- Xcode project: `NearbyFinder.xcodeproj`, targets `NearbyFinder` (iPhone-only: `SUPPORTED_PLATFORMS = iphoneos iphonesimulator`, `TARGETED_DEVICE_FAMILY = 1` — iPad/Mac/Vision have no UWB) and `NearbyFinderWatch` (watchOS companion, embedded with `platformFilter = ios`)
- Deployment target: iOS/watchOS 26.0 (lowered from the 26.5 template default for release reach); bundle IDs `jp.hibiki.treasureradar` / `.watchkitapp` (renamed pre-release; `WKCompanionAppBundleIdentifier` must stay in sync); iPhone is locked to portrait
- The `#if os(iOS)` / `#else` stubs in the managers are dead code on the only remaining platform but kept as guards. Note `canImport(NearbyInteraction)` is NOT a sufficient platform check — the module imports on macOS with unavailable APIs.
- No test targets yet. No Swift Package dependencies.

## Commands

Build for iOS Simulator (Debug):

```sh
xcodebuild -project NearbyFinder.xcodeproj -scheme NearbyFinder \
  -destination 'platform=iOS Simulator,name=iPhone 17' build
```

Build the watch app alone: `-scheme NearbyFinderWatch -destination 'generic/platform=watchOS Simulator'`.

Run tests (once a test target is added): replace `build` with `test`; run a single test by appending `-only-testing:<TestTarget>/<TestClass>/<testMethod>`.

There is no linter configured.

## Architecture

Layering, bottom to top:

- `MultipeerSession.swift` — thin MC wrapper, service type `nearbyfinder`, peer displayName `"\(deviceName)#\(random4digits)"` (iOS 16+ reports "iPhone" for every device; the suffix disambiguates). Connection strategy (each rule exists because of a shipped failure — keep all of them): (1) normally only the peer with the larger `displayName` invites, avoiding simultaneous-invitation handshake collisions; (2) if still unconnected ~6 s after discovering the peer, the other side also invites, because MC discovery is often one-directional and a designated-inviter-only scheme deadlocks; (3) a ~2 s jittered retry loop re-invites while unconnected, rebuilds the whole transport (fresh peer ID/session/advertiser/browser) after 3 consecutive failures since MC can wedge internally, and re-kicks browsing every ~12 s while nothing is discovered (also on foreground return via `refreshDiscoveryIfNeeded()`). Invitations are rejected while already connected (no double sessions). `stop()` must rebuild a fresh transport after teardown — torn-down advertiser/browser have their delegates detached and can't be restarted; this is what makes タイトルへ → はじめる and the lobby's 接続をやり直す button work. `isDesignatedLeader(vs:)` is exposed so exactly one side performs connection-time work (settings broadcast).
- `NearbySessionManager.swift` — `ObservableObject` owning the player `NISession`; publishes `status`, `distance`, `direction`, and `horizontalAngle`. On iPhone 14 and later `NINearbyObject.direction` is ALWAYS nil (`supportsDirectionMeasurement == false` — Apple removed UWB-only direction), so the `HuntingView` arrow falls back to `horizontalAngle`, which camera assistance provides only after ARKit convergence (bright scene, user sweeps the phone). Don't remove that fallback and don't treat nil `direction` on modern devices as a bug. Token exchange is the simple form: on MC connect each side sends `.discoveryToken(Data)` and runs `NINearbyPeerConfiguration` on receipt (the sessionID/ACK protocol lives only in the archive branch). Camera assistance (`isCameraAssistanceEnabled`, requires `NSCameraUsageDescription`) uses NI's INTERNAL ARSession — deliberately no `setARSession`/shared session (see rebuild note). Permission denial (`.userDidNotAllow`) must NOT restart the session — it would loop; camera denial falls back to no assistance. `start()`/`stop()` lifecycle serves the title screen: `isStarted` guards re-entry (re-start just refreshes discovery), `stop()` stops MC, invalidates the player and watch-peer NI sessions, and resets published state. `onConnected(isLeader:)` fires on MC connect (leader = larger displayName) for the settings broadcast. Also relays watch tokens (`.watchToken`/`.watchPeerToken`) and game state to the paired Watch via `relayGameStateToWatch`.
- `GameManager.swift` — game state machine (`lobby → hiding → hunting → finished`) plus `ProximityFeedback` (distance-driven haptic pulses synced with sonar pings). Roles sync over `.roleSelected` (a random `priority` breaks the both-picked-the-same-role tie). Either player's one tap starts BOTH devices — a deliberate no-host design (2 players in the same room negotiate verbally; don't add a host/ready-check flow). The abruptness on the passive side is softened by `RoleRevealView`: `isRevealingRole` covers the first ~2 s of hiding on both devices with an「あなたは○○」announcement — keep it. (A `.chase` mode existed briefly — 逃走中 — but was removed as unwanted; it exists only in git history a952ccc. Don't propose re-adding it.) Hide/hunt durations are lobby-configurable (in-memory only) and synced over `.settingsChanged` (last-write-wins); `.roleSelected` also carries the initiator's settings, which are authoritative for that round. Because settings can diverge (a relaunched device resets to defaults), the designated leader broadcasts its settings once on connection (`nearby.onConnected`) so both lobbies match. Settings are editable ONLY in the lobby — a title-screen settings sheet was added and removed once (the user doesn't want the same settings on two screens); don't reintroduce it. The hunt has a time limit (`huntDuration`, default 300 s); the *treasure* device is authoritative for timeout — its watchdog sends `.timeUp` and both finish with `outcome = .treasureWon`. Rounds form a best-of series (`seriesTarget` = first to 3): round wins are counted locally from (outcome, own role) so both devices agree without extra messages, and `.rematch` swaps roles and starts the next round (guarded by `phase == .finished`, which makes simultaneous rematch taps converge). Returning to the lobby resets the series. Found is confirmed physically, not by distance: the hunter uses the `SlideToConfirmButton` on the *treasure* device's screen — a phone-call-answer-style slider with no hold delay (`confirmFound()`, treasure-side only), which sends `.found`. The near-full-width directional slide is the accidental-touch protection; don't replace it with a tap or long-press (fabric/skin contact triggers those), and don't add a hold-to-arm delay back (tried; felt unresponsive). Distance-based auto-detection was removed deliberately — proximity fires before the phone is physically located. Win counts and best clear time (`GameStats`) live in memory for the current launch only — the app DELIBERATELY persists nothing (no UserDefaults/files anywhere): zero required-reason APIs means no `PrivacyInfo.xcprivacy` is needed and the App Store privacy story is "no data stored". If you ever add persistence, you must also add a privacy manifest declaring the user-defaults required-reason API (CA92.1) or the upload gets ITMS-91053. During hunting it records `distanceHistory` (1 sample/sec, both devices) for the Swift Charts approach graph in `ResultView`, and relays phase/role/deadline/outcome to the paired Watch via `syncWatch()` on every transition. It re-publishes the nested `NearbySessionManager`'s `objectWillChange` — nested `ObservableObject`s don't propagate automatically.
- `GameAudio.swift` — sounds are synthesized at runtime with AVAudioEngine (sine-wave ping / fanfare / time-up buffers); there are no audio asset files. Uses the `.ambient` session category on purpose: respects the silent switch and mixes with other audio.
- Views: `ContentView.swift` (title screen + phase switch + `LobbyView`/`HidingView`/`TreasureWaitView`/`ResultView`) and `HuntingView.swift` (Find-My-style finding UI: black background, big arrow driven by `arrowAngle` — `direction` on U1-era iPhones, `horizontalAngle` fallback on iPhone 14+ — remaining-time header, green fade + pulse rings under 1 m). `TitleView` gates everything: `game.start()` (radios, permission prompts) and `isIdleTimerDisabled = true` only run when the user taps はじめる — don't move them back to launch. The lobby's タイトルへ button reverses this via `game.stop()`; the lobby also has a 接続をやり直す button (`restartConnection()`) while searching/connecting. Start is disabled via `nearby.isDeviceSupported` (checkable before `start()`, false in the stub).
- Apple Watch support (`NearbyFinderWatch/`, hunter-side accessory display): watchOS has no MultipeerConnectivity, so tokens relay Watch → WatchConnectivity (`PhoneWatchRelay` on iOS, applicationContext keys `watchToken`/`peerToken`) → own iPhone → MC (`.watchToken`/`.watchPeerToken` messages) → other iPhone, which opens a SECOND `NISession` (`watchPeerSession`, minimal `WatchPeerSessionDelegate`) against the watch and replies with its token. UWB ranging then runs directly Watch ↔ other iPhone; `WatchRanger` publishes distance (watchOS NI provides distance only, never direction) and `WatchHuntView` renders it with wrist haptics. Game state (phase/role/deadline/outcome) is relayed in the same applicationContext — note `updateApplicationContext` REPLACES the whole dictionary, so `PhoneWatchRelay` keeps one merged `outgoingContext` (token + state) and always sends it whole. The watch hides distance outside the hunting phase and shows the outcome with a haptic when the game ends. The watch app suspends on wrist-down — ranging resumes on wrist raise.

## Constraints and conventions

- The build uses `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` (Xcode 26 template). Framework delegate callbacks (MCSessionDelegate, NISessionDelegate) arrive on non-main queues, so delegate conformances are declared `nonisolated` and hop to the main actor via `Task { @MainActor in ... }`. Follow this pattern for any new delegate-based code.
- Info.plist: `GENERATE_INFOPLIST_FILE = YES` is combined with a checked-in `Info.plist` at the repo root (`INFOPLIST_FILE = Info.plist`), which holds the keys that can't be expressed as `INFOPLIST_KEY_*` build settings — notably the `NSBonjourServices` array (`_nearbyfinder._tcp/_udp`, required for MC browsing) plus the NI/local-network/camera usage descriptions and `ITSAppUsesNonExemptEncryption`. It lives *outside* the `NearbyFinder/` folder on purpose: that folder is a `PBXFileSystemSynchronizedRootGroup`, and a plist inside it would be bundled as a resource and conflict.
- NI peer sessions suspend when either app leaves the foreground; both phones must keep the app frontmost.
